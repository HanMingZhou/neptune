# 开发规范（补充约定）

> 本文件是 `project_rules.md` 的补充，记录项目中积累的具体约定和踩坑经验。

---

## 1. 路由与接口约定

- 所有路由以 `/api/v1` 为前缀，后接业务模块名称
- 除明确需要 GET 的接口外，统一使用 POST
- URL 格式示例：`/api/v1/operation/record/add`

## 2. 请求与响应模型

- Service 层的入参 `req` 和返回值 `resp` 结构体必须定义在 `model/request/` 和 `model/response/` 中
- 禁止直接将 DB Model 作为接口返回值

## 3. 数据库模型管理

- DB Model 与其他 Model 通过 ID 关联
- 新增 Model 后必须在 `server/initialize/gorm.go` 中注册 AutoMigrate

## 4. 工具函数

- 工具函数统一放在 `server/utils/` 对应子包中

## 5. 文档与总结

- 除非用户明确要求，不要自动生成总结文档或 Markdown 文件

## 6. 列表页通用规范

- 所有列表页必须包含刷新按钮，使用全局统一的 `RefreshButton` 组件
- 列表数据每 10 秒自动刷新一次

## 7. 前后端数据约定

- 后端除动态菜单外，不返回 icon 和 color 字段，由前端自行维护
- 前后端禁止使用 mock 数据，未完成的功能统一提示「功能开发中」

## 8. 动态菜单

- 前端菜单通过接口动态获取并渲染（菜单名称、菜单 icon），即动态路由模式

## 9. 后端代码风格

- 注释统一使用中文
- 使用常量代替魔法值，常量定义在对应功能模块的 `model` 层

## 10. 国际化（i18n）规范

- 在 Vue 页面中使用 `t('key')` 时，必须同步在 `web/src/i18n/locales/zh-CN.json` 和 `en-US.json` 中添加对应词条
- `t()` 函数在找不到翻译时会原样返回 key，容易造成「以为是英文模式」的假象，务必及时补全翻译
- 禁止复制粘贴时引入重复的 `<script setup>` 块

## 11. UnoCSS 与 Element Plus 样式冲突

- UnoCSS Preflight 会通过标签选择器重置样式，可能覆盖 Element Plus 的组件样式
- 项目中使用 `@layer components` 包裹 Element Plus 覆盖样式，但 `@layer` 优先级低于非 layer 样式
- 解决方案：在全局 CSS（`index.css`）中使用 `!important` 强制覆盖，这是 UnoCSS + Element Plus 混合项目的标准做法
- 新增 Element Plus 组件样式被覆盖时，参照 `index.css` 中已有的修复模式处理
